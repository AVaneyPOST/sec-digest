## General high-severity

- **Gogs unpatched 0-day — argument injection → RCE via crafted PR branch name** — Rapid7 published the advisory on the pre-announced 2026-05-28 disclosure date after maintainers went silent following 2026-03-28 acknowledgement. Affects Gogs 0.14.2 and 0.15.0+dev (i.e. latest). Exploit: create a malicious branch name that injects `--exec` into `git rebase`, then trigger "Rebase before merging." Technically authenticated, but Gogs default config ships with **open registration and no repo-creation limit** — so an attacker just signs up. No patch available. Internet-facing default Gogs should be assumed compromisable on demand; pull behind auth/VPN or move to Gitea immediately. [BleepingComputer](https://www.bleepingcomputer.com/news/security/new-gogs-zero-day-flaw-lets-hackers-get-remote-code-execution/) · [The Hacker News](https://thehackernews.com/2026/05/critical-gogs-rce-vulnerability-lets.html) · Date: 2026-05-28 · **HIGH**

- **Gitea CVE-2026-27771 — container registry served private images without auth** (CVSS 8.2) — Access-control bug: `private` flag on container repositories was never enforced at the registry endpoint, so anonymous `docker pull` against a known private repo returned manifests and layers. Sat undetected ~4 years. ~30k internet-exposed Gitea instances impacted across 30+ countries (CN/US/DE/FR/UK lead), including healthcare providers, aerospace, retail, ISPs. Risk = leaking app code, API keys, DB creds, internal infra config inside layers. Fix: upgrade to Gitea **v1.26.2**; interim mitigation is `[service].REQUIRE_SIGNIN_VIEW=true` (kills all anonymous access including intended-public). Discovered by Noscope's autonomous pentest agent. [The Hacker News](https://thehackernews.com/2026/05/gitea-vulnerability-exposes-private.html) · [Noscope advisory](https://www.noscope.com/blog/gitea-instances-exposing-private-container) · [SecurityWeek](https://www.securityweek.com/gitea-vulnerability-exposed-30000-deployments-to-attacks/) · Date: 2026-05-28 · **HIGH**

- **FortiClient EMS CVE-2026-35616 actively weaponised — EKZ infostealer delivered as fake Fortinet hotfix** — Arctic Wolf published the campaign report on 2026-05-28. The CVE itself (unauth API access-control bypass, CVSS 9.1, patched in 7.4.5/7.4.6 on 2026-04-04) is now being chained into a credential-stealing operation: attacker hits unpatched EMS, pushes a payload disguised as a Fortinet endpoint update, executes via PowerShell across all EMS-managed endpoints. The EKZ stealer pulls Chrome creds via `IElevator::DecryptData` (v20 AES-256 master key bypass) and Firefox creds by dynamically loading `nss3.dll` against `key4.db`/`logins.json`/`cookies.sqlite`. ~2k EMS instances were exposed at original disclosure; many remain unpatched. [Arctic Wolf — EKZ campaign writeup](https://arcticwolf.com/resources/blog/forticlient-ems-exploited-via-cve-2026-35616-to-deliver-ekz-infostealer-disguised-as-a-fortinet-patch/) · [BleepingComputer](https://www.bleepingcomputer.com/news/security/hackers-exploit-forticlient-ems-flaw-to-push-infostealer-malware/) · [The Hacker News](https://thehackernews.com/2026/05/threat-actors-exploit-critical.html) · Date: 2026-05-28 · **HIGH**

- **Carnival Corp data breach disclosure — ~6M individuals, ShinyHunters claim** — Carnival filed disclosure on 2026-05-27 (notifications began same day); BleepingComputer confirmed scale and ShinyHunters attribution on 2026-05-28.
