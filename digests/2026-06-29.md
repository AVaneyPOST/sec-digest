## Telecom

- **Cisco Unified Communications Manager — CVE-2026-20230 (SSRF→file write→RCE) KEV
  deadline 2026-06-28 expired** *(KEV add 18 Jun, observed exploitation 21–22 Jun,
  deadline 28 Jun, HIGH — telecom voice)*
  The "patch by Sunday" deadline from BOD 26-04 fell yesterday. UCM is the Cisco
  enterprise VoIP control plane (SIP registrar, CTI, call routing); the SSRF in
  WebDialer can be chained to write arbitrary files to the filesystem with the
  service account's privileges, and operator practice has been to either patch to
  the fixed maintenance trains or disable WebDialer (which is off by default in
  fresh installs but enabled in many migrated estates). The relevant telco-team
  question for Monday is whether *every* PBX-replacement and contact-center estate
  inherited the disabled-by-default posture rather than a legacy "always on"
  config. [Cisco PSIRT cisco-sa-cucm-webdialer / CISA KEV
  <https://www.cisa.gov/known-exploited-vulnerabilities-catalog>] Date: deadline
  28 Jun 2026. [HIGH]

- **PTC Windchill PDMLink / FlexPLM — CVE-2026-12569 (improper input validation →
  RCE) KEV deadline 2026-06-28 expired** *(CVSS 9.3, observed exploitation week of
  22 Jun, JSP webshells confirmed by vendor, HIGH — adjacent to telecom-OEM
  engineering supply chain)*
  PTC's advisory was last updated 25 Jun warning of *continued* "heightened threat
  activity"; that update is still the most recent vendor statement at the start of
  the new week. The exploitation pattern is JSP webshell drop on the Windchill /
  FlexPLM application server, then arbitrary command execution and exfiltration of
  PLM artifacts (BoMs, CAD, supplier files). For telecom-vendor and operator
  engineering supply chains this is the relevant pivot — Windchill is the data
  repository where RAN / core / handset hardware designs live at Tier-1 vendors
  and several MNOs. [PTC trust center
  <https://www.ptc.com/en/about/trust-center/advisory-center/active-advisories/windchill-flexplm-rce-vulnerability>

## General high-severity

- **DirtyClone — CVE-2026-43503 — Linux kernel LPE-to-root** *(public PoC 25 Jun,
  weekend coverage cycle 27–28 Jun, CVSS 8.8, HIGH — Linux kernel)*
  Local-to-root via two helper functions in the skb-clone path
  (`__pskb_copy_fclone()` and `skb_shift()`) dropping a flag that marks page-cache
  pages as MAP_SHARED-backed; an attacker walks a setuid binary's pages into a
  cloned skb, routes it through an attacker-controlled IPsec tunnel, and the
  decrypt step overwrites the in-memory copy of `/usr/bin/su` (or any chosen
  binary) with attacker bytes. The on-disk binary is unchanged → file-integrity
  monitoring tools see nothing; standard auditd / syslog see nothing because there
  is no file syscall. JFrog Security Research published the working walkthrough on
  25 Jun. Mainline fix landed 21 May. Default-config Debian and Fedora are
  exploitable because user namespaces are unprivileged. Practitioner action:
  confirm your rolling-kernel and long-term-support trees both carry the May-21
  fix; do not rely on FIM alerts for detection.
  [<https://thehackernews.com/2026/06/new-dirtyclone-linux-kernel-flaw-lets.html>
  / JFrog blog] Date: 25 Jun 2026. [HIGH]

- **Polymarket frontend supply-chain — loss revised to $3.1M (26–28 Jun reporting)**
  *(NOTABLE — supply chain / web frontend / crypto)*
  Disclosure date is 25 Jun; the in-window development is the loss-figure update
  to $3.1M and the on-chain tracing work by PeckShield / SpecterAnalyst / GoPlus
  Security through Sat 27 Jun, plus the related CFTC-investigation coverage on
  Sun 28 Jun. Mechanism: a third-party vendor that supplies Polymarket's
  frontend was compromised, attacker pushed malicious JS that quietly prompted
  some users to sign wallet-drain transactions; smart contracts on Polygon were
  not touched. The pattern (third-party JS dependency → frontend → user-asset
  drain) is the same shape an operator self-care portal, eSIM provisioning
  storefront, or carrier billing UI inherits whenever it embeds vendor-supplied
  JS. [<https://www.securityweek.com/3-million-reportedly-stolen-in-polymarket-hack/>
  / <https://decrypt.co/372129/polymarket-refund-users-scammers-swipe-millions-website-exploit>]
  Date: revised loss reporting 27–28 Jun 2026. [NOTABLE]

  (430k FortiGates, ~110M credentials, Russian-speaking IAB, FortigateSniffer
