## Telecom

- **FortiBleed — credential & config leak for 73,932 Fortinet FortiGate / SSL VPN devices across 194 countries, disclosed 2026-06-17 [HIGH, telecom-heavy].** Researcher Volodymyr "Bob" Diachenko surfaced the dataset; Hudson Rock and SOCRadar published analyses on 2026-06-17, with Arctic Wolf, TechCrunch, Hackread, and BleepingComputer covering same day. Attribution: a multi-operator Russian-speaking cybercriminal group. Methodology: large-scale credential reuse (older Fortinet dump credentials + infostealer logs replayed against internet-exposed FortiGate on 443), SSL VPN auth-hash interception cracked offline on a 45-GPU Hashtopolis cluster, and on-device watching for fresh creds once inside. Scale numbers: ~1.16B credential attempts vs ~320,000 FortiGate targets; 30,791 confirmed working creds in their working DB; 21,632 unique compromised domains. Per Hudson Rock and SOCRadar geo distribution: India, US, Taiwan, Mexico lead. **Telecom-sector exposure is heaviest (>5,600 entries by SOCRadar count)**, with Comcast and AT&T explicitly named in coverage alongside Samsung, Foxconn, Oracle, Siemens, Lenovo, PwC, Accenture; multiple government agencies and critical-infrastructure operators are also said to be present. Practitioner relevance: many Tier-1/Tier-2 MNOs and MVNOs run FortiGate SSL VPN as either primary remote-access edge or for partner / roaming-engineer access; appearance in this dataset is consistent with full pre-auth network access and almost certainly downstream creds harvested on-device. Action: (1) check Hudson Rock's free FortiBleed lookup or SOCRadar's checker against every fqdn you own that fronts a FortiGate, (2) rotate all device admin and VPN-user creds for any matched device, (3) force MFA on SSL VPN if you haven't already, (4) audit SSL-VPN session logs back six months and treat any unexpected admin-portal access on 443 as IR-worthy, (5) hunt for the on-device persistence/exfil indicators Hudson Rock and SOCRadar publish. ## General high-severity

- **Mastra npm supply chain — 144 @mastra/* packages backdoored via easy-day-js typosquat, ~1.1M weekly downloads of exposure (Socket / StepSecurity / The Hacker News, 2026-06-17) [HIGH].** Compromise vector: hijacked legitimate-contributor "ehindero" npm account whose scope access had not been revoked. Attack window: ~88 minutes after publishing the typosquat payload at easy-day-js@1.11.22, the actor used the org credentials to mass-republish 144 packages with `"easy-day-js": "^1.11.21"` quietly added as a production dependency — semver-caret resolution means every fresh `npm install` automatically pulls 1.11.22. Payload chain: obfuscated postinstall dropper → pull-and-execute → self-delete → second-stage cross-platform infostealer / RAT (Phoenix.Security and SafeDep call out RAT behaviour; Socket emphasises the infostealer) targeting browser history and 160+ cryptocurrency-wallet browser extensions, installing persistence across Windows / macOS / Linux, and exfiltrating to attacker C2. @mastra/core is ~918k weekly downloads on its own; combined exposure >1.1M weekly. GitHub recommended downgrading Mastra pending package removal. Practitioner relevance: telecoms running internal LLM-agent stacks built on Mastra — and anyone shipping AI tooling to NOC / SOC operators — need to (1) audit lockfiles for any `@mastra/*` versions published within the 88-minute window, (2) `npm audit signatures` or use Socket / StepSecurity / SafeDep advisory feeds to enumerate, (3) revoke and rotate any credentials touched on dev workstations during the install window, (4) treat as a host-compromise event on machines that ran `npm install` since 2026-06-17.
