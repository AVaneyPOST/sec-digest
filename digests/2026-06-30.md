# Security digest — 2026-06-30

Window: ~24h ending 2026-06-30 04:00 UTC. Telecom-focused. Tuesday morning — the
in-window story is the **public PoC drop for libssh2 CVE-2026-55200** (a critical
client-side SSH RCE that's bundled into half the embedded telecom estate), plus the
0DIN "Clone This Repo" indirect-prompt-injection attack on AI coding agents. The
Cisco Catalyst SD-WAN Manager **CVE-2026-20262 KEV deadline expired yesterday
(Mon 29 Jun)** under BOD 26-04. No new verified GSMA / SS7 / Diameter / N32 advisory
inside the window, and no new CISA KEV add since the 25 Jun pair.

## TL;DR

- **libssh2 CVE-2026-55200 — public PoC released Mon 29 Jun.** Critical CVSS 9.2
  client-side memory corruption: a malicious SSH server can drive an out-of-bounds
  heap write in any client that links libssh2 ≤ 1.11.1 — no auth, no user
  interaction. The cleanup pain is the long tail of *bundled and static copies*
  inside curl, Git, PHP, backup agents, firmware updaters and embedded appliances
  — which for telecoms is the eNB/gNB management plane, OAM gateways and
  lawful-intercept appliances that ship with vendor-shipped libssh2 nobody can
  trivially repackage. The patch landed in mainline 12 Jun and the CVE was
  published 17 Jun; the new variable today is the public exploit harness.
  [HIGH — embedded / telecom-OEM supply chain]
- **Cisco Catalyst SD-WAN Manager CVE-2026-20262 KEV deadline expired Mon 29 Jun.**
  Arbitrary file write, actively exploited by Cisco PSIRT's own statement; the
  BOD-26-04 two-week clock ran out yesterday. For any MNO running Catalyst SD-WAN
  under a carrier-managed-WAN or MEC umbrella, the assumption that the deadline
  pass = remediation done has to be verified per-controller, not per-policy.
  [HIGH — telecom managed-WAN / MEC]
- **0DIN "Clone This Repo" — indirect prompt-injection → reverse shell on the
  developer's box (Mon 29 Jun).** A benign-looking GitHub repo whose README and
  setup script chain through a DNS-TXT-resolved init command and `curl | bash`,
  triggered when an AI coding agent (Claude Code, Cursor, etc.) walks the
  developer through the project's "init" flow. The repository itself contains no
  overtly malicious code; the payload is fetched at runtime, so code review,
  static analysis and the agent's own read of the repo see nothing. Same shape
  reaches anyone whose telco-tooling repos are public or whose internal repos
  pull from public mirrors. [NOTABLE — supply chain / AI-agent tooling]
- **Hijacked npm + Go packages → VS Code task → Python infostealer.** New
  Hacker News write-up Mon 29 Jun: two npm packages and a cluster of Go packages
  drop a `.vscode/tasks.json` that auto-runs on folder open, pulls encrypted JS
  out of blockchain transaction data, opens a socket.io backdoor, then deploys
  a wide-spectrum credential / wallet / dev-artifact stealer (Git creds, gh
  hosts.yml, VS Code global storage, Keychain, KDE Wallet, etc.). Same vector
  reaches a telco operator's automation / IaC repos as cleanly as any other dev
  shop. [NOTABLE — supply chain / dev tooling]
- **No new GSMA / SS7 / Diameter / GTP / N32 / SEPP advisory verified in window.**
  No new CISA KEV add since the 25 Jun pair (PTC Windchill CVE-2026-12569 +
  Cisco UCM CVE-2026-20230). CISA's Vulnerability Summary SB26-180 published
  29 Jun is the routine weekly roll-up for the week of 22 Jun.

## Telecom

- **Cisco Catalyst SD-WAN Manager — CVE-2026-20262 (arbitrary file write,
  actively exploited) KEV deadline expired Mon 29 Jun** *(CVSS 6.5, in-the-wild
  exploitation confirmed by Cisco PSIRT June 2026, HIGH for SD-WAN-managed MEC /
  mobile-backhaul estates)*
  SD-WAN Manager (formerly vManage) is the control plane for many operator MEC,
  mobile-backhaul and managed-CPE fabrics; the file-write primitive chains
  cleanly to root on the controller (one of the public write-ups walks the
  primitive through to RCE). Cisco's first-fixed-release guidance is per branch
  and there is no workaround that fully addresses the exposure without upgrading.
  The deadline pass under BOD 26-04 is yesterday's event; the operator-side
  question for today is whether the patch was applied to *every* tenant /
  carrier-managed instance versus only to the federally-mandated ones.
  [SecurityAffairs <https://securityaffairs.com/193693/security/cve-2026-20262-cisco-catalyst-sd-wan-flaw-under-active-targeted-exploitation.html>
  / SocRadar <https://socradar.io/blog/cve-2026-20262-cisco-catalyst-sd-wan-manager/>
  / CISA KEV] Date: deadline 29 Jun 2026. [HIGH]

- **libssh2 — CVE-2026-55200 — public PoC released Mon 29 Jun** *(CVSS 9.2,
  client-side OOB heap write via crafted server packets, HIGH — embedded
  telecom / OAM supply chain)*
  See "General high-severity" for the full write-up. The reason this earns a
  Telecom-section line as well: libssh2 is the SSH-client stack of choice in a
  long list of telecom-vendor management and OAM products (eNB/gNB EMS connectors,
  lawful-intercept appliances, OSS/BSS connectors, firmware-update agents on
  CPE/HPE/OLT estates). When the SSH *client* is the vulnerable side, the threat
  model flips: an attacker who can stand up a hostile SSH endpoint that one of
  these connectors reaches into — including a poisoned jump host or a wedged
  vendor-cloud bastion — can land code in the operator's management plane
  without ever touching an exposed listening port. Practitioner action: catalogue
  every appliance and vendor agent that *initiates* SSH on your management VRFs
  and demand a bundled-libssh2 statement from each vendor before treating this as
  patched. [The Hacker News / VulnCheck advisory (CVE published 17 Jun)
  <https://cvedaily.io/2026/06/29/critical-libssh2-client-side-flaw-cve-2026-55200-poc-released/>]
  Date: PoC publication 29 Jun 2026. [HIGH]

- **Ericsson Packet Core Gateway — security bulletin June 2026 (CVE-2026-25657,
  CVE-2026-25658, CVE-2026-25659)** *(DoS / service degradation via malformed
  signalling messages, HIGH — telecom core)*
  Three improper-handling defects in the PCG message parser: a malformed
  syntactically-invalid structure (CVE-2026-25657) and two missing-values cases

  inside June and remains the most recent PCG advisory from Ericsson; I could
  not independently verify the *exact* publication day inside the 24h window
  through public channels (Ericsson's PSIRT page is the primary source), so
  this is included with a date caveat. [Ericsson PSIRT
  <https://www.ericsson.com/en/about-us/security/psirt/security-bulletin-packetcoregateway-june-2026>]
  Date: June 2026 (exact day unverified inside window). [HIGH — telecom core, date unverified]

## General high-severity

- **libssh2 — CVE-2026-55200 — public PoC released Mon 29 Jun** *(CVSS 9.2,
  CWE-680 integer-overflow-to-buffer-overflow, HIGH)*
  Root cause: `ssh2_transport_read()` fails to bound-check the `packet_length`
  field on incoming SSH packets; an attacker supplies an over-large length, the
  multiplication overflows, and the subsequent allocation/copy lands as an
  out-of-bounds heap write the client never re-validates. The PoC harness
  published 29 Jun in the *exploitarium* GitHub archive is described by its
  author as a "locally verified SSH trigger scaffold and a controlled local RCE
  harness" rather than a turnkey one-shot — the gap between the harness and a
  reliable wild exploit is measured in days, not weeks. No credentials, no user
  interaction; the only precondition is that the client *connects* to the
  attacker's server (or to any server the attacker can position MITM against
  for a libssh2 connection that doesn't pin the server). Affected: libssh2 ≤
  1.11.1; patch merged via PR #2052 on 12 Jun and CVE published by VulnCheck on
  17 Jun. The real cleanup cost is finding bundled and static copies inside
  curl, Git, PHP, backup agents, firmware updaters, network appliances and
  embedded systems — a vendor-by-vendor inventory exercise, not a
  package-manager pass. [The Hacker News
  <https://thehackernews.com/2026/06/public-poc-released-for-critical.html>
  / cvedaily summary
  <https://cvedaily.io/2026/06/29/critical-libssh2-client-side-flaw-cve-2026-55200-poc-released/>
  / cybersecuritynews
  <https://cybersecuritynews.com/libssh2-vulnerability/>]
  Date: PoC publication 29 Jun 2026. [HIGH]

- **Hijacked npm + Go packages → VS Code task → Python infostealer (Hacker News
  write-up Mon 29 Jun)** *(NOTABLE — supply chain / dev tooling)*
  Two npm packages and a cluster of Go packages, uploaded to npm 25 May 2026
  and pulled by the time of disclosure, ship a `.vscode/tasks.json` configured
  to auto-execute when the project folder is opened in VS Code. The task fetches
  encrypted JavaScript stored inside on-chain blockchain transaction data,
  decrypts and runs it client-side, opens a socket.io backdoor to attacker
  infrastructure, then drops a Python-based infostealer with a deliberately
  developer-targeted lens: Git credentials, GitHub CLI `hosts.yml`, GitHub
  Desktop logs, VS Code workspace and global storage, Windows Credential
  Manager, Linux Secret Service, KDE Wallet, macOS Keychain, plus cloud-storage
  metadata. Practitioner action: grep your repos for unreviewed `.vscode/tasks.json`
  with `runOptions.runOn: folderOpen`; treat any folder-open task that shells
  out to an unverified script as compromised until proven otherwise. The
  blockchain-transaction-data-as-payload-carrier pattern is the one that breaks
  network-level DLP. [The Hacker News
  <https://thehackernews.com/2026/06/hijacked-npm-and-go-packages-use-vs.html>]
  Date: 29 Jun 2026 write-up. [NOTABLE]

- **Mozilla 0DIN — "Clone This Repo and I Own Your Machine" — indirect prompt
  injection of AI coding agents (Mon 29 Jun)** *(NOTABLE — supply chain / AI
  agents)*
  Mozilla's Zero Day Investigative Network published a PoC showing a malicious
  GitHub repo with no overtly malicious code can compromise a developer
  machine through any AI coding agent (Claude Code, Cursor, etc.) the
  developer invokes on the repo. The chain: a benign README and a Python
  package engineered to fail on first use → the failure message instructs the
  user (and therefore the agent) to run an "init" command → that command
  resolves a DNS TXT record under the attacker's control and pipes the record
  contents to bash. The reverse-shell payload is fetched at runtime and never
  lives in the repo, so static analysis, code review and the agent's own
  reading of the repo all see nothing. The shell runs as the developer's
  user, with the developer's tokens, cloud creds and SSH keys. Operator
  relevance: any internal Claude-Code / Cursor / GH-Copilot deployment that
  walks engineers through "clone and try it" workflows against vendor or
  customer-supplied repos. [Help Net Security
  <https://www.helpnetsecurity.com/2026/06/29/mozilla-warns-of-indirect-prompt-injection-risk-in-ai-coding-agents/>
  / SecurityWeek
  <https://www.securityweek.com/new-attack-abuses-claude-code-and-harmless-looking-repositories-to-hijack-developer-machines/>
  / 0DIN advisory <https://0din.ai/blog/clone-this-repo-and-i-own-your-machine>]
  Date: 29 Jun 2026. [NOTABLE]
